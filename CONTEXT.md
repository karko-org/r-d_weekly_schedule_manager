# 프로젝트 컨텍스트 & 이력

## 🎬 프로젝트 배경

### 시작점
사용자는 울산 KAR 기술연구소의 자동차 R&D 엔지니어로, 매주 월요일 아침 브리핑 미팅에서 개인 업무 일정을 공유해야 합니다. 기존에는 동료가 공유해준 엑셀 양식(`5_최종민_주임_업무내용.xlsx`)을 매주 수작업으로 업데이트했습니다.

엑셀 양식 구조:
- **Sheet1**: 연간 간트 (사용 안함)
- **Sheet2**: 월간 캘린더 (우리가 구현한 포맷)
  - 가로축: 일자 (1~31)
  - 세로축: 월 블록마다 "요일 행 + 업무 행들"
  - 다중 일자 업무는 셀 병합으로 표현

### 왜 웹앱으로?
이전에 성공적으로 만든 "KAR 이메일 서명 생성기"가 단일 HTML 도구로 잘 작동했고, 사용자는 같은 패턴을 원했습니다. **팀 대시보드가 아니라 개인 도구** 임을 명확히 했습니다.

## 🔄 버전 히스토리

### v1.0 — 초기 프로토타입
- 일일 시간표 형식으로 시작 (x축 시간, y축 태스크)
- **피드백**: "월간 캘린더가 필요해. 시간표 아니고."

### v1.1 — 월간 캘린더로 전환
- 원본 엑셀 Sheet2 양식 재현
- 간트 행 자동 배치 알고리즘 (`assignRows`)

### v1.2 — 엑셀 양방향 I/O
- SheetJS로 import/export
- 셀 병합 감지 파서
- 섹션 분리: 이번주 + 전체 (접기/펼치기)

### v1.3 — 브리핑 모드 강화
- 다크 테마 브리핑 뷰
- 지난주/이번주 토글 추가 (`briefingWeek = 'last' | 'this'`)

### v1.4 — In-place 파일 저장 (중요한 UX 전환)
- File System Access API 도입
- **계기**: "매번 다운로드 받기 싫어. 원본 파일에 바로 저장하고 싶어"
- IndexedDB (idb-keyval)로 파일 핸들 영속화 → 브라우저 재시작 후에도 연결 유지

### v1.0 (현재) — 세부 기능 다수 추가
사용자 요청별로 증분 개선:
1. ✅ 이번주/다음주 토글 (입력 뷰)
2. ✅ `새 파일로 내보내기` 파일명 자동 생성 (`{부서}_{이름}.xlsx`)
3. ✅ 부서 기본값 `기술연구소`
4. ✅ 이번주 항목 인라인 편집 (`✎` 버튼)
5. ✅ 박스 밖 클릭 = 편집 모드 자동 종료 (focusout)
6. ✅ 액션바 sticky (스크롤해도 상단 유지)
7. ✅ 자동 열기 폴더 기억 (`startIn: fileHandle`)
8. ✅ 10개월 샘플 데이터 + 연도 롤오버 파서 (12 → 1 자동 감지)
9. ✅ 브리핑 모드 스크롤 버그 수정 (`body.briefing-active{overflow:hidden}` 제거)
10. ✅ 이번주/다음주 섹션 접기 (기본 접힘)
11. ✅ `새 파일로 내보내기` 버튼 스타일 통일 (아이콘 없는 `btn-secondary`)
12. ✅ **캘린더 클릭 → 상세 모달** (전체 업무 섹션으로 이동 대체)
13. ✅ 상세 메모 (notes) 필드 + 엑셀 셀 코멘트로 영속화
14. ✅ 브리핑 모드에서도 상세 모달 열기
15. ✅ 상세 모달에 `저장` 버튼 (닫기 대체, 파일 저장 X)
16. ✅ 캘린더 가독성 토글 (기본/일정 펼치기)
17. ✅ 펼치기 모드에서 텍스트 중앙 정렬 + 오버플로우 버그 수정

## 🎨 설계 결정사항 (Why/Why-Not)

### 왜 단일 HTML 파일?
- 사용자가 회사 PC (Edge) + 개인 PC (Chrome)에서 모두 써야 함
- 빌드 도구 없이 바로 수정/배포 가능
- USB/이메일로 공유 쉬움
- ~87KB로 가볍고 CDN 로딩만 필요

### 왜 React가 아닌 Vanilla JS?
- 단일 파일 유지 목표
- 상태 복잡도가 낮음 (월별 업무 배열 + 뷰 상태 2~3개)
- 빌드 파이프라인 필요 없음

### 왜 File System Access API?
- 대안 1 (매번 다운로드): 사용자가 싫어함
- 대안 2 (서버 동기화): 과한 인프라, 보안 이슈
- 결론: 원본 엑셀 파일에 직접 저장이 가장 자연스러움
- **트레이드오프**: Chrome/Edge만 지원 → 사용자가 두 브라우저 모두 가능하므로 OK

### 왜 IndexedDB?
- File handle은 LocalStorage에 저장 불가 (직렬화 안됨)
- `idb-keyval` CDN으로 간단히 영속화

### 왜 셀 코멘트로 notes 저장?
- 별도 시트/파일 관리 복잡함
- 엑셀에서 열어도 셀 우측 상단 삼각형으로 확인 가능
- 다른 컴퓨터에서 파일 열어도 메모 유지

### 연도 롤오버 처리 (중요!)
**문제**: 사용자가 10개월 데이터 (2025.8 ~ 2026.5) 테스트 시 2026.4가 2025.4로 로딩되는 버그 발생.

**원인**: 파서가 엑셀 제목의 `YYYY년`을 한 번만 읽고 모든 월에 동일 연도 적용.

**해결** (v1.0):
```javascript
let curYear = year;
let prevMonth = null;
monthStarts.forEach(ms => {
  // 월 번호가 이전 월보다 작아지면 연도 롤오버 감지
  if (prevMonth !== null && ms.month < prevMonth) curYear += 1;
  // ...
  prevMonth = ms.month;
});
```

## 🐛 주요 버그 수정 이력

### 1. 브리핑 스크롤 불가 (해결됨)
- **증상**: 브리핑 뷰에서 업무 10개 넘어가면 화면 하단 잘림
- **원인**: `body.briefing-active { overflow: hidden }` 규칙이 브리핑 스크롤도 차단
- **해결**: 해당 CSS 규칙 제거. `#input-view { display: none }`만으로 충분.

### 2. 연도 인식 실패 (해결됨)
- **증상**: 2026년 데이터가 2025년으로 로딩
- **원인**: 단일 연도로 모든 월 처리
- **해결**: 월 순회 중 롤오버 감지 로직 추가

### 3. 편집 중 포커스 손실 (해결됨)
- **증상**: 이번주 항목에서 날짜 입력 중 renderThisWeek 호출되어 커서 사라짐
- **해결**: 편집 중에는 tw-list 전체 재렌더링 X, 해당 항목의 날짜 텍스트만 업데이트

### 4. 캘린더 펼치기 모드 오버플로우 (해결됨, 최근)
- **증상**: 긴 텍스트가 옆 바까지 침범 (이미지로 증명됨)
- **원인**: `overflow: visible` + `justify-content: center` 조합 → 텍스트가 바 경계 밖으로 확장
- **해결**: `overflow: visible` 제거 (베이스 `overflow: hidden` 상속), `overflow-wrap: anywhere` 추가, 최소 너비 1200 → 1400px

## 📊 데이터 모델

### 메모리 구조 (`monthData`)
```javascript
{
  "2026-04": {
    tasks: [
      {
        id: "t_abc123",     // uid() 생성
        title: "TM 설계",
        start: 20,          // 해당 월의 날짜 (1-31)
        end: 24,
        notes: "세부 메모"  // 선택적
      }
    ]
  }
}
```

### LocalStorage 키
- `kar_monthly_schedule_v1` — monthData
- `kar_weekly_writer_v1` — { name, dept }
- `kar_excel_file_handle_v1` — IndexedDB로 저장, file handle
- `kar_cal_view_v1` — `'basic' | 'expanded'`
